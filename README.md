# EvoTravel 🌍

> 会学习的 AI 旅行规划助手——它记得你的偏好，自动规划最优路线，在地图上实时展示行程，让每一次出行都更懂你。

## ✨ 特性

- **自然语言交互** — 不需要搜索，直接说"我想去杭州玩两天，从上海出发"，AI 自动理解意图并规划
- **智能偏好记忆** — 跨旅程记住你的偏好（节奏、口味、兴趣），用得越多规划越准
- **实时地图展示** — 高德地图 API 在地图上标注所有地点，展示最优游览路线
- **智能路线优化** — 自动计算最优游览顺序、通勤方式、时间估算
- **同城多分店智能选择** — 自动选择离上一站最近的分店，不会把楼外楼标到崇明岛
- **跨城交通规划** — 12306 列车查询 + LLM 车次推荐
- **酒店推荐与锚点** — 基于行程中心点搜索酒店，自动设为住宿锚点
- **通勤方案卡片** — 地铁/公交换乘方案、打车费用估算

## 🛠 技术栈

| 层级 | 技术 |
|------|------|
| 前端 | 原生 HTML/CSS/JavaScript（零框架依赖） |
| 地图 | 高德地图 JS API + Web 服务 API |
| AI | 智谱 GLM-5（结构化 JSON 输出） |
| 交通 | 12306 列车查询 + 高德公交换乘 API |
| 数据 | localStorage（偏好/历史持久化） |

## 🧠 技术实现思路

### LLM 驱动决策，而非正则猜测

传统方案用正则匹配用户意图和提取信息，容易误判且难以扩展。EvoTravel 将意图识别、城市提取、地点替换全部交给 GLM-5 输出结构化 JSON，前端只做字段解析，保证语义理解的准确性。

### 结构化输出协议

GLM 规划行程时，在末尾输出 `## 📍 地点列表`（按天分组）。前端只解析这个固定格式，普通聊天回复不含此标记则不触发地图和路线计算。一个标记就解决了"什么时候该规划 vs 什么时候只是聊天"的判断问题。

### 地点校验反馈循环

地理编码后校验坐标是否在目标城市内。不在目标城市的地点 → 让 GLM 推荐同城同类型替代 → 重新校验 → 最多 3 轮循环，解决 LLM 幻觉导致的地点错误。

### 同城多分店最近优先

POI 搜索返回多个同城结果时，基于上一站坐标计算距离，自动选择最近的分店。

### 偏好记忆系统

每轮对话后台提取用户偏好（节奏、口味、兴趣类型等），持久化到 localStorage。"新旅程"按钮清空对话但保留偏好，形成使用粘性。

## 🚀 Agent 进化路径

**L0 规则助手** — 正则匹配意图，固定模板回复。已经历并淘汰。

**L1 LLM 对话** — 接入大模型做自然语言对话，但无法操控外部服务。已经历。

**L2 结构化 Agent** ✅ — LLM 输出结构化数据，前端解析后调用地图/POI/交通 API，形成"感知 → 规划 → 执行"闭环。

**L3 自校正 Agent** ✅ — 地理编码不匹配时自动反馈 LLM 重新推荐，多轮校验保证准确性。

**L4 记忆 Agent** ✅ — 跨会话偏好记忆，用户用得越多规划越准。

**L5 主动 Agent** 🔄 — 心跳轮询，主动推送天气预警、行程提醒、价格变动。

**L6 多 Agent 协作** 🔮 — 酒店/交通/餐饮 Agent 各司其职，主 Agent 编排调度。

## 💰 业务价值

- **个性化体验** — 记住用户偏好，每次规划都不一样，形成使用粘性
- **降低门槛** — 从"搜索"到"对话"，不需要知道该搜什么
- **自动化优化** — 路线、通勤、时间全自动计算
- **可扩展** — 接入预订 API 佣金变现，偏好数据精准推荐，企业版差旅规划

## 🚀 快速开始

1. 克隆仓库
```bash
git clone https://github.com/Jackie-Li-0228/evotravel.git
cd evotravel
```

2. 用任意 HTTP 服务器打开（高德地图需要 HTTP 环境加载）
```bash
# 例如用 Python
python3 -m http.server 8080
```

3. 浏览器打开 `http://localhost:8080`

> 需要配置自己的高德 API Key 和智谱 API Key（在 `app.js` 和 `prompt.js` 中）
